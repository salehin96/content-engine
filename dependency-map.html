SEO Content Engine · Dependency Map

What each phase actually reads, and why

Seven phases run in order, brand foundation through article images. Each box shows only the inputs that phase consumes directly, not everything downstream inherits. Nodes are colored by kind so an engine file and a tool never look alike, and each carries a badge for how much of it is read.

Node kinds

Engine file a repo file the engine produced
Tool an MCP, service, or CLI
External input the live web / research
Gate must exist, not read as content
full reads the whole file part reads only named sections gate presence required, content not consumed out produced by the phase, not read

Reading notes

Shared rule files are omitted on purpose. Every phase also loads the same governance set, research-integrity rules, tool routing, fallback behavior, agent dispatch patterns, and the data schemas. They govern how a phase runs, not what it is about, so they are left off the nodes to keep each phase's real data inputs legible.

The fetch ladder shown as one tool is really a fallback chain: WebFetch, then Firecrawl, then Claude in Chrome (Reddit fetched with a .json suffix), then ask the user to paste.

Lifecycle and inventories are review-owned. The topical map's status columns, the entity inventory, and the linking-anchor inventory are written by the orchestrator and the review phase, never by the phase that merely reads them.
